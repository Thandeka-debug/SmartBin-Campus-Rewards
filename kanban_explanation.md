# Kanban Board Explanation

## My Kanban Board Columns

My SmartBin Sprint Board uses the following columns, built from previous assignments:

| Column | Purpose |
|--------|---------|
| **Backlog** | All tasks from Assignment 6 product backlog |
| **Blocked** | Tasks that cannot proceed due to dependencies |
| **Ready** | Tasks fully defined and ready to be worked on |
| **In Progress** | Tasks currently being worked on (WIP limit: 3) |
| **Testing** | Completed code awaiting quality assurance (WIP limit: 2) |
| **In Review** | Code that has passed testing and is awaiting final review |
| **Done** | Fully completed and verified tasks |

## Custom Columns

The assignment required adding 2+ custom columns. My board includes:
- **Blocked** column - Added in Assignment 6 to track dependencies
- **Testing** column - Added in Assignment 6 for QA tracking

These columns were retained and customized for Assignment 7.

## Work-in-Progress (WIP) Limits

| Column | WIP Limit | Justification |
|--------|-----------|---------------|
| In Progress | 3 tasks | Single developer cannot work on more than 3 tasks simultaneously |
| Testing | 2 tasks | Testing requires focus; too many indicates quality issues |
| In Review | 2 tasks | Code review requires attention; limits prevent bottleneck |
| Blocked | No limit | Blocked tasks need visibility but shouldn't count against WIP |

## How the Board Supports Agile Principles

| Agile Principle | How My Board Supports It |
|----------------|--------------------------|
| Continuous delivery | Tasks move through columns to Done |
| Adaptability | Easy to reprioritize by moving tasks between columns |
| Transparency | Everyone can see what is Blocked, In Progress, or In Review |
| Customer focus | Each task links to a user story from Assignment 6 |
